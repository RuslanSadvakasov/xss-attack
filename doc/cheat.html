Правило №1

Никогда не доверяй пользовательскому вводу. Любые данные из req.body, req.query, localStorage, URL — потенциально опасны.

📋

Input Validation

Проверка данных ДО обработки. Решает: принимать или отклонять.

✅ ЧТО ДЕЛАТЬ:

  • • Белые списки — разрешай только ожидаемые символы
  • • Проверяй тип данных (number, email, uuid)
  • • Ограничивай длину строки
  • • Библиотеки: validator.js, Joi, Zod, Yup

❌ ЧЕГО НЕ ДЕЛАТЬ:

  • • Чёрные списки — всегда можно обойти
  • • Попытки "починить" данные вместо отказа
  • • Только фронтенд-валидация — легко обходится
✅ email@example.com → ПРОПУСТИТЬ
❌ <script>alert(1)</script> → ОТКЛОНИТЬ
🔄

Output Encoding

Преобразует данные в безопасный формат, чтобы браузер не воспринимал их как код.

HTML Body &lt; → &lt;
HTML Attribute &quot;
JavaScript \xHH \uXXXX
URL encodeURIComponent() 
// ОПАСНО: "<script>alert(1)</script>"
// БЕЗОПАСНО: "&lt;script&gt;alert(1)&lt;/script&gt;"
// → браузер покажет как ТЕКСТ, не выполнит

🎯 Контекст решает всё!

HTML, атрибуты, JS, CSS, URL — для каждого своё кодирование

✅

Safe Sinks

Безопасные места для данных — не выполняют код.

textContent
innerText
setAttribute()
className
value
createTextNode()
⚠️

Dangerous Sinks

Опасные места — интерпретируют данные как код.

innerHTML
document.write()
eval()
setTimeout(string)
insertAdjacentHTML
onclick + innerHTML
🎯

Контекстное экранирование

Один и тот же символ в разных контекстах нужно экранировать по-разному. Выбор метода зависит от того, куда вы вставляете данные.

</>

HTML Body

Между тегами HTML

< → &lt;

Пример: &lt;script&gt;

⚠️ Без экранирования: <script> выполнится

🔗

URL / URI

Параметры ссылок

%HH encoding

Пример: %22 %3C %3E

⚠️ Без экранирования: javascript: атака

📜

JavaScript String

Внутри JS кода

\uXXXX Unicode

Пример: \u003C \u003E \u0022

⚠️ Без экранирования: "); alert(1); // выполнится

🎨

CSS String

Внутри CSS правил

\XX HEX encoding

Пример: \3C \3E \22

⚠️ Без экранирования: expression(alert(1)) (старые IE)

📊 Один и тот же символ — разное экранирование

Символ HTML URL JavaScript CSS
" &quot; %22 \u0022 \22
< &lt; %3C \u003C \3C
> &gt; %3E \u003E \3E
& &amp; %26 \u0026 \26

⚠️ ПОЧЕМУ НЕЛЬЗЯ ИСПОЛЬЗОВАТЬ ОДИН МЕТОД ДЛЯ ВСЕХ КОНТЕКСТОВ? 

// Разработчик сделал только HTML-экранирование
element.textContent = userInput;  // ✅ безопасно для HTML

// Но забыл про JS-контекст!
<script>
    var data = "ПОЛЬЗОВАТЕЛЬСКИЕ_ДАННЫЕ";
    // Если userInput = "\"; alert(1); //"
    // HTML-экранирование НЕ СПАСЁТ! Нужно JS-экранирование \uXXXX
</script>
💡

Золотое правило

Всегда выбирай метод экранирования под конкретный контекст, куда вставляются данные. HTML-экранирование не защитит в JS, а URL-экранирование — в CSS.

📚 Библиотеки для безопасного экранирования: escape-html, he, lodash.escape

🧼

HTML Sanitization

Удаляет опасный HTML (<script>, event handlers), но сохраняет безопасное форматирование (<b>, <i>, <a>).

📦 DOMPurify — лучшая библиотека

import DOMPurify from 'dompurify';

const dirty = '<b>Hello</b><script>alert(1)</script>';
const clean = DOMPurify.sanitize(dirty);
// Результат: '<b>Hello</b>'

container.innerHTML = clean; // теперь безопасно!
✗ удаляет <script>
✗ удаляет onclick=
✗ удаляет javascript:
✓ сохраняет <b>, <i>

Дополнительные защиты

Защита Что делает Пример настройки
CSP Блокирует inline-скрипты и загрузку с недоверенных доменов CSP: script-src 'self'
Trusted Types Требует явного создания безопасных строк для dangerous sinks require-trusted-types-for 'script'

Правильные примеры кода

✅ Безопасное добавление текста в DOM

// ❌ ОПАСНО
div.innerHTML = userComment;

// ✅ БЕЗОПАСНО
div.textContent = userComment;

// ✅ ЕЩЁ БЕЗОПАСНЕЕ
const textNode = document.createTextNode(userComment);
div.appendChild(textNode);

✅ Безопасная работа с атрибутами

// ❌ ОПАСНО (можно внедрить javascript:)
link.href = userInput;

// ✅ БЕЗОПАСНО
link.setAttribute('href', userInput);

// ✅ С ДОПОЛНИТЕЛЬНОЙ ПРОВЕРКОЙ
if (!userInput.startsWith('javascript:')) {
    link.href = userInput;
}
✅

Чек-лист перед релизом

▸ Все пользовательские данные проходят валидацию
▸ Экранирование применяется ПЕРЕД выводом
▸ Отсутствует innerHTML с внешними данными
▸ Настроен CSP (хотя бы report-режим)
▸ Сессионные cookie имеют флаг HttpOnly
▸ DOMPurify для пользовательского HTML
▸ Валидация дублируется на сервере
▸ Безопасные DOM API вместо innerHTML
